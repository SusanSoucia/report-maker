## 一、实验任务

本次实验基于 Philippe Kruchten 提出的 4+1 视图模型，对 ATM 自动存取款机软件系统进行架构分析与设计。实验要求在 4+1 视图框架下完成以下工作：

1. **场景视图**：分析系统参与者，建立用例模型，绘制核心业务的时序图，覆盖正常流程与异常流程；
2. **逻辑视图**：进行面向对象分析与设计，提取核心类并定义类间关系，采用合适的设计模式；
3. **开发视图**：划分软件包/模块，建立模块依赖关系，定义目录组织结构；
4. **进程视图**：划分进程/线程，描述并发机制与线程交互流程，设计并发控制策略；
5. **物理视图**：规划系统部署拓扑，包括 ATM 终端、应用服务器集群、数据库、安全硬件等节点的映射关系。

此外，在 4+1 视图的基础上，对系统的接口设计、数据结构、运行机制、安全性、异常处理及性能扩展等方面进行设计分析，并通过需求追踪矩阵验证设计对需求的覆盖完整性。

## 三、实验内容

### 3.1 体系结构风格与总体设计

#### 3.1.1 风格选择

ATM 系统选择**分层架构（Layered Architecture）**作为顶层体系结构风格，理由如下：

| 考量维度 | 分析 |
|----------|------|
| 关注点分离 | ATM 系统的硬件驱动、交易逻辑、网络通信、用户界面属于明显不同的关注域，分层自然对应这些边界 |
| 硬件依赖隔离 | 读卡器、出钞机等硬件变更不能影响上层交易逻辑——分层将硬件隔离在最底层，上层仅依赖抽象 |
| 通信复杂性 | 与银行核心系统间的 ISO 8583 报文通信需要独立封装，分层将网络层隔离 |
| 可测试性 | 各层可独立 mock 测试：不依赖真实硬件即可验证交易逻辑正确性 |

对比了**微内核架构**（硬件驱动可插拔，但 ATM 硬件类型有限，过度设计）和**事件驱动架构**（适合状态流转但不适合硬件指令的顺序调用），最终选择分层架构。

#### 3.1.2 分层设计

| 层次 | 包名 | 职责 |
|------|------|------|
| 界面层 | `atm.ui` | ATM 主控制器、屏幕显示、键盘输入，管理 ATM 状态流转 |
| 交易层 | `atm.transaction` | 交易处理调度、交易策略执行、并发控制 |
| 账户层 | `atm.account` | 账户验证、余额管理、账户实体 |
| 硬件抽象层 | `atm.hardware` | 读卡器、出钞机、存款槽、凭条打印机的统一抽象 |
| 网络通信层 | `atm.network` | 与银行核心系统 TLS 通信、连接池管理、请求重试 |
| 安全模块 | `atm.security` | 身份认证、PIN 验证、数据加解密、哈希 |
| 审计日志 | `atm.logging` | 异步日志记录，所有交易操作可追溯 |

#### 3.1.3 设计模式选型

| 设计模式 | 应用场景 | 选择理由 |
|----------|----------|----------|
| **State（状态）** | ATM 操作状态流转 | ATM 的生命周期（空闲→插卡→验证→菜单→交易→退卡）是典型的状态机，State 模式使状态转换规则集中管理，新增状态不扰动现有状态逻辑 |
| **Strategy（策略）** | 不同交易类型调度 | 取款、转账、存款、查询四种交易的操作逻辑差异大，Strategy 模式让 TransactionProcessor 在运行时切换执行策略，符合开闭原则 |
| **Template Method（模板方法）** | 交易执行框架 | 所有交易共享同一流程骨架（验证→预授权→执行→确认→日志），差异步骤由子类实现，避免重复代码 |
| **Singleton（单例）** | 硬件控制器、网络模块、日志模块 | 硬件资源全局唯一，多实例会导致设备冲突；网络连接池和日志队列同理 |
| **Proxy（代理）** | 银行核心系统通信 | NetworkModule 作为 BankCoreSystem 的远程代理，封装 TLS 加密、连接管理、超时重试，使上层模块对网络细节透明 |
| **Observer（观察者）** | 硬件状态变更通知 | 出钞机缺钞、打印机缺纸等事件需要及时通知 UI 和运维模块，Observer 模式解耦事件源和响应者 |
| **Facade（外观）** | ATMController 对外接口 | ATMController 作为整个 ATM 子系统的外观，外部（如银行监控系统）只需调用统一接口而不关心内部复杂性 |

每个模式的选择都直接回应了需求中的约束条件——例如 State 模式回应了"交易状态转换设计"（8.5 节）的需求，Strategy 模式回应了"系统功能模块划分"（3.4 节）中交易类型的多样性要求。

#### 3.2.3 取款业务流程

取款是 ATM 最高频的核心业务。图 3-2 展示了完整的取款时序，包含身份验证、预授权、出钞、确认扣款四个阶段，以及身份验证失败、预授权失败（余额不足）、出钞失败三条异常分支。

![图 3-2 取款业务时序图](image/sequence_withdraw.png)

**设计要点：**
- 采用**预授权-确认-撤销**三段式交易模型：先申请预授权（资金冻结），出钞成功后再确认扣款，出钞失败则撤销预授权。保证资金不会凭空消失或凭空出现
- 出钞指令在预授权通过后才发出，避免"扣了钱没出钞"的事故
- 所有关键节点（验证、预授权、出钞、确认）均记录审计日志，满足需求 9.11 的审计要求

#### 3.2.4 转账业务流程

图 3-3 展示了转账业务的时序，与取款类似采用预授权-确认二阶段模型，但涉及转出和转入两个账户的操作。

![图 3-3 转账业务时序图](image/sequence_transfer.png)

**设计要点：**
- 转账操作对一致性要求最高——不能出现"转出扣了款但转入没收到"的情况。采用预授权阶段对双方账户同时校验，确认阶段原子性执行
- 预授权失败（余额不足、目标账户无效）在早期返回明确错误码，避免用户等待超时

#### 3.3.3 关键设计决策

**为什么 Transaction 是抽象类而非接口？**
Transaction 有共享的状态（accountId、amount）和模板方法骨架（execute 调用 validate → doExecute → logTransaction）。抽象类适合"部分实现+部分抽象"的场景，接口则用于完全抽象的契约（如 TransactionStrategy）。

**为什么 ConcurrencyManager 独立于 TransactionProcessor？**
并发控制逻辑（锁获取/释放、重试策略、退避算法）是横切关注点，与具体交易逻辑正交。独立为 ConcurrencyManager 使两者都可独立测试和演进。

**version 字段放在 Account 上（而非 TransactionRecord 上）？**
乐观锁的冲突检测粒度是"同一账户的并发写操作"。将 version 放在 Account 上，任何修改该账户余额的操作都会触发冲突检测——这是正确的粒度。如果放在 TransactionRecord 上，只能检测同一条记录的并发，覆盖不了不同交易操作同一账户的场景。

#### 3.4.1 包划分原则

开发视图关注系统的模块化组织。包划分遵循**高内聚低耦合**原则，按功能层次垂直切分：

### 3.5 进程视图 —— 并发与运行设计

#### 3.5.1 进程/线程划分

ATM 终端的进程视图关注运行时的并发结构。系统采用多线程模型，划分如下：

| 线程 | 数量 | 职责 | 优先级 |
|------|------|------|--------|
| 主线程（事件循环） | 1 | ATMController 状态机、UI 响应、交易分发 | 高 |
| 交易工作线程池 | 3 | 并发执行交易（取款/转账/存款） | 正常 |
| 网络 I/O 线程池 | 4 | 异步银行通信、连接管理 | 正常 |
| 硬件监控守护线程 | 1 | 定期巡检硬件状态（每 5s） | 低 |
| 审计日志异步线程 | 1 | 异步写入日志文件 | 低 |

#### 3.5.3 并发控制设计

多线程环境下，以下场景存在并发竞争：

| 场景 | 风险 | 策略 |
|------|------|------|
| 同一账户同时取款和转账 | 余额计算错误（丢失更新） | **乐观锁**：Account.version 字段，更新时 `WHERE version = oldVersion`，冲突则重试 |
| 出钞机并发访问 | 多个交易争抢出钞指令 | HardwareController 内部使用 **synchronized** 方法串行化出钞操作 |
| 网络连接池竞争 | 线程饥饿 | 连接池上限（含等待超时） |
| 日志写入竞争 | 写冲突 | AuditLogger 内部使用 **BlockingQueue**，单线程消费 |

**选择乐观锁（而非悲观锁）的理由：**
- ATM 场景下账户冲突概率低（同一账户很少同时在不同 ATM 操作）
- 悲观锁（`SELECT ... FOR UPDATE`）会长时间持锁——ATM 交易涉及网络往返，持锁等待银行响应将严重降低并发度
- 乐观锁的重试成本低——重新读取最新余额、校验、更新，通常在 100ms 以内完成

**重试策略**：最多 3 次，指数退避（100ms → 200ms → 400ms），超过 3 次返回"交易繁忙，请稍后重试"。

#### 3.6.3 关键部署决策

**为什么应用服务器 3 台？** 2 台可保证单点故障时仍有服务，3 台提供更好的负载分担和滚动发布能力。对本科生实验场景，3 是成本和可靠性的合理平衡点。

**为什么数据库主从分离？** 余额查询是高频操作（占交易量 60% 以上），将只读查询路由到从库可显著降低主库压力，提升写操作（取款/转账）的响应速度。

**ATM 终端离线模式**：当网络不可达时，终端缓存交易请求并在恢复后批量重放。每个请求带幂等键（idempotency_key），保证重放不会重复扣款。

| Account | `version` | 乐观锁版本号，每次更新 +1 |
| Account | `pin_hash` | PIN 不存明文，使用 bcrypt + salt 哈希存储 |
| TransactionRecord | `idempotency_key` | 幂等键，由 atm_id + timestamp + sequence 组成，防止交易重复 |
| TransactionRecord | `status` | 状态枚举：pending→authorized→confirmed/cancelled/failed，完整跟踪交易生命周期 |
| AuditLog | `severity` | 分级：INFO（正常操作）、WARN（重试/降级）、ERROR（失败需关注）、CRITICAL（安全事件） |

### 3.8 安全设计

#### 3.8.1 安全层次

| 层次 | 机制 | 防护目标 |
|------|------|----------|
| 身份认证 | PIN + bcrypt 哈希验证 | 防止非法用户操作账户 |
| 传输安全 | TLS 1.3，全链路加密 | 防止中间人窃听、篡改 |
| 数据安全 | 敏感字段加密存储，HSM 密钥管理 | 防止数据库泄露导致 PIN 外泄 |
| 报文安全 | HSM 签名 + 完整性校验（HMAC-SHA256） | 防止交易报文被篡改 |
| 防重放 | 幂等键 + 时间窗口（60s） | 防止重放攻击导致重复扣款 |
| 访问控制 | 管理员操作需双因素认证 | 防止未授权的加钞/维护操作 |
| 审计 | 全操作日志记录，不可删除 | 事后追溯与合规 |

#### 3.8.2 连续密码错误处理

PIN 连续错误 3 次 → 锁定卡片（Card.status = 'locked'）→ 吞卡 + 告警通知银行。锁定后需持卡人本人到柜台验证身份后解锁。（回应需求 9.9 节）

### 3.9 异常处理设计

#### 3.9.1 异常分类与策略

| 异常类别 | 典型场景 | 处理策略 | 用户感知 |
|----------|----------|----------|----------|
| 可恢复异常 | 网络超时（偶发） | 自动重试 3 次，指数退避 | 无感知 |
| 可补偿异常 | 出钞失败 | 撤销预授权，资金回滚 | "交易已撤销" |
| 需人工介入 | ATM 断电恢复 | 从持久化日志恢复状态，标记待处理 | "请稍候…" |
| 致命异常 | 数据库不可达 | 进入离线模式，缓存请求 | "网络不可用" |

#### 3.9.2 关键异常流程

**出钞失败 → 撤销预授权**（已在图 3-2 取款时序图中体现）：
1. 出钞机返回故障 → HardwareController 通知 TransactionProcessor
2. TransactionProcessor 调用 NetworkModule 向银行发起撤销请求（携带原始预授权 authCode）
3. 银行确认撤销 → 资金回滚到账户
4. 记录 AuditLog（severity=ERROR），通知 UI 显示"设备故障，交易已撤销"
5. HardwareController 标记出钞机不可用，触发 Observer 通知运维

**网络超时 → 重试 + 幂等保证**（已在图 3-5 异常活动图中体现）：
1. 请求超时 → NetworkModule 重试（最多 3 次）
2. 每次重试携带相同 idempotency_key
3. 银行端通过 idempotency_key 识别重复请求，返回原结果而非重复执行
4. 3 次重试均失败 → 进入离线模式

**ATM 断电恢复**（回应需求 10.11 节）：
1. 重启后检查 TransactionRecord 中 status='pending' 或 'authorized' 的记录
2. 向银行查询每笔交易的真实状态
3. 已授权未确认的交易 → 自动撤销（超时补偿）
4. 已确认的交易 → 打印凭条（补打）
5. 状态不一致的 → 标记为待人工处理，告警通知

### 3.12 设计总结

#### 3.12.1 实验完成情况

本次实验完整覆盖了 ATM 系统中 4+1 视图的五个维度，共：
- 用例 7 个，覆盖 3 类参与者
- UML 图 10 张（用例图、取款时序图、转账时序图、系统活动图、异常处理活动图、类图、包图、进程交互图、部署图、ER 图）
- 设计模式 7 个，每个均有需求对应和选型理由
- 需求追踪覆盖功能需求 23 项、非功能需求 6 项，覆盖率 100%

#### 3.12.2 大模型辅助完成的内容

本报告的主要文字撰写和 PlantUML 代码生成由 Claude（大语言模型）辅助完成。其中：
- 类图和时序图初稿由模型生成，人工进行了类关系校验和参数调整
- 需求追踪矩阵由模型根据章节内容自动生成，人工核对覆盖完整性
- 所有 PlantUML 代码保留在 `image/` 目录中，可独立运行

#### 3.12.3 人工检查与修改的内容

- TransactionProcessor 与 TransactionStrategy 的关系从实现（`..|>`）修正为持有/依赖（`-->`），符合 Strategy 模式中 Context 持有 Strategy 引用而非实现 Strategy 接口的正确定义
- 时序图中出钞失败异常分支的撤销流程补充了银行确认撤销的返回步骤
- Account 实体增加 version 字段以支持乐观锁
- 部署图中补充了 HSM 和日志中心节点

#### 3.12.4 实验收获

1. 4+1 视图不是五张孤立的图——它们从不同视角描述同一系统，一致性是最大挑战。本实验通过需求追踪矩阵和一致性检查表系统性地保证了五个视图的协调。
2. 设计模式的选择必须有需求依据。"用了什么模式"不重要，"为什么这个模式解决了这个需求"才重要。本次实验中每个模式都回溯到具体需求条目。
3. 并发控制方案的选择不存在银弹——乐观锁在低冲突场景最优，单账户高频操作则需要降级为悲观锁或分布式锁。理解场景比记住方案更重要。
